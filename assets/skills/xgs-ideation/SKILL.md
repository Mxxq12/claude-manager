# XGS Ideation — 深度头脑风暴

将想法通过深度思维分析和协作对话，转化为清晰的结论或可执行的方案。

## 场景判断

头脑风暴启动后，先判断话题类型：

- **开发类** — 需要出设计文档和实施计划（走完整流程）
- **探索类** — 调研、选型、趋势分析等，不一定需要文档输出（走轻量流程）
- **创意类** — 产品创意、商业想法、方案对比等（按需决定输出）

根据类型灵活调整流程深度和输出形式。

<HARD-GATE>
如果是开发类话题：设计未经用户批准前，禁止写任何实现代码、创建项目脚手架、或调用任何实现类 skill。
</HARD-GATE>

## Checklist

你必须为以下每一步创建 task 并按顺序完成（根据场景类型可跳过不适用的步骤）：

1. **了解背景** — 探索相关上下文和已有信息
2. **提问理解需求** — 可一次问多个相关问题，理解目的/约束/成功标准
3. **深度思维分析** — 根据话题类型选用合适的思维框架组合
4. **提出 2-3 个方案** — 含权衡分析和推荐
5. **展示结论/设计** — 按复杂度分段展示，逐段获得用户确认
6. **输出文档（如需要）** — 开发类保存到 `docs/specs/YYYY-MM-DD-<topic>-design.md`，其他类型按需决定
7. **用户确认后过渡（如需要）** — 开发类询问是否进入实施计划，同意后调用 `xgs-plan` skill

## 流程详述

### 1. 了解背景

- **开发类：** 查看项目文件结构、文档、最近 git 提交，了解现有架构和技术栈
- **探索类：** 搜索相关信息、行业动态、已有方案
- **创意类：** 了解用户的背景、目标、已有的想法和约束

### 2. 提问理解需求

- **可以一次问多个相关问题**，不必拘泥于一次一问
- 优先用选择题降低回答成本，必要时用开放式问题
- 关注：目的、受众、约束条件、成功标准、优先级
- 在问细节之前先评估范围：如果涉及多个独立主题，先帮用户拆分，再逐个深入

### 3. 深度思维分析

这是本 skill 的核心。根据话题类型和复杂度，从以下工具箱中**智能选择合适的组合**：

#### 思维工具箱

**5W2H 系统拆解**
- Why — 为什么要做？不做会怎样？
- What — 具体做什么？边界在哪？
- Who — 谁是受众？谁参与？谁受影响？
- When — 时间约束？阶段划分？
- Where — 在哪里实施？影响哪些范围？
- How — 具体方案？实现路径？
- How much — 成本？工作量？资源预算？

**假设挑战**
- 列出所有隐含假设（技术假设、业务假设、用户假设、认知假设）
- 逐一质疑：这个假设真的成立吗？如果不成立会怎样？
- 识别高风险假设，提出验证方法

**逆向思维（Pre-mortem）**
- 假设这件事已经失败了，最可能的原因是什么？
- 从失败原因反推需要提前解决的问题
- 识别最大风险点和缓解策略

**用户/角色场景推演**
- 定义关键角色或利益相关方
- 从不同角色的视角走一遍流程
- 关注边缘情况和容易被忽略的场景
- 思考可能的误用或意外情况

**发散-收敛双阶段**
- **发散阶段：** 不评判、不筛选，尽可能多地产出方案和想法
- **收敛阶段：** 按标准筛选，合并相似方案，排列优先级

**多维度评估矩阵**
- 可行性（能力匹配度、资源是否充足）
- 价值（解决痛点的程度、影响范围）
- 成本（时间、金钱、精力投入）
- 风险（不确定性、依赖、潜在问题）
- 可扩展性（未来变化的适应能力）

#### 工具选用指南

| 话题类型 | 推荐工具组合 |
|---------|------------|
| 新功能开发 | 5W2H + 场景推演 + 发散-收敛 |
| 架构设计/重构 | 5W2H + 假设挑战 + 逆向思维 + 多维评估 |
| Bug 修复方案 | 假设挑战 + 逆向思维 |
| 技术选型 | 多维评估 + 假设挑战 |
| 产品探索/MVP | 5W2H + 场景推演 + 逆向思维 + 发散-收敛 |
| 性能优化 | 假设挑战 + 多维评估 |
| 调研/趋势分析 | 5W2H + 发散-收敛 + 多维评估 |
| 商业/战略决策 | 5W2H + 假设挑战 + 逆向思维 + 多维评估 |
| 创意探索 | 发散-收敛 + 场景推演 + 假设挑战 |
| 问题诊断 | 假设挑战 + 逆向思维 + 场景推演 |

以上为默认推荐，可根据实际情况灵活调整组合。

### 4. 提出 2-3 个方案

- 每个方案包含：核心思路、优缺点
- 明确推荐其中一个并说明理由
- 对比维度根据话题调整（开发类：复杂度、可维护性、性能；商业类：ROI、风险、时间线等）

### 5. 展示结论/设计

- 按复杂度分段展示：简单部分几句话，复杂部分 200-300 字
- 每段展示后询问用户确认
- **开发类覆盖：** 架构、组件、数据流、错误处理、测试策略
- **探索类覆盖：** 关键发现、推荐方向、下一步行动
- **创意类覆盖：** 核心概念、可行路径、关键风险

### 6. 输出文档（如需要）

根据话题类型决定是否输出文档：

**开发类（必须）：**
- 保存到 `docs/specs/YYYY-MM-DD-<topic>-design.md`
- 提交到 git
- 自审清单：占位符？矛盾？范围合理？模糊需求？
- 交用户审查

**探索类/创意类（按需）：**
- 如果用户需要，整理成文档保存
- 如果不需要，口头总结即可

### 7. 用户确认后过渡（如需要）

**开发类：**
> "设计已确认。是否现在进入自动执行？"
- 用户同意 → 调用 `xgs-autopilot` skill，全自动完成计划制定、实现、测试、交付
- 用户拒绝 → 停止

**探索类/创意类：**
> "头脑风暴结束。还有什么想继续深入的吗？"

绝不自动跳转，必须等用户明确指示。

## 关键原则

- **深度优先** — 宁可在思考阶段多花时间，也不要匆忙下结论
- **渐进验证** — 分段展示，逐步确认，不要最后扔一大坨内容
- **智能适配** — 思维工具和流程深度要匹配话题类型，不是每次都全部用上
- **挑战假设** — 主动质疑看似理所当然的东西
- **用户主导** — 每个阶段过渡都需要用户确认，不自动跳转
- **灵活输出** — 不是所有头脑风暴都需要文档，按需决定输出形式
