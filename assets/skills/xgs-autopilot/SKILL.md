# XGS Autopilot — 自动驾驶执行

方案确认后，全自动完成从计划到交付的全流程。

## 核心理念

**用户已经在头脑风暴阶段做了决策，剩下的交给我。**

- 自己写计划、自己执行、自己测试、自己调试、自己验证
- 遇到问题自己排查修复，不甩给用户
- 只在真正需要用户决策时才问

## 什么时候可以问用户

**可以问：**
- 不可逆操作（删除数据、付费服务、推送到远程仓库）
- 方案方向性变更（发现设计有根本缺陷，需要改方向）
- 缺少必要的凭证或权限

**不要问：**
- 技术实现细节（自己判断）
- 遇到 bug（自己调试）
- 测试失败（自己修）
- 依赖冲突（自己解决）
- "这样行不行？" — 行就做，不行就换方案
- "要不要我继续？" — 当然继续，别停

## 全流程

### 阶段 1：制定计划

1. 基于确认的设计方案，拆分为具体任务
2. 每个任务明确：要改哪些文件、代码怎么写、怎么测试
3. 任务粒度：每步 2-5 分钟，一个动作
4. TDD 流程：写测试 → 确认失败 → 写实现 → 确认通过 → 提交
5. 计划保存到 `docs/plans/YYYY-MM-DD-<feature-name>.md`
6. **不问用户确认，直接进入执行**

### 阶段 2：执行实施

对每个任务：

1. 按计划步骤逐一执行
2. 写测试、跑测试、写代码、跑测试、提交
3. 每完成一个任务自动进入下一个
4. **不在任务之间停下来问用户**

**使用子代理加速：**
- 独立任务可以派子代理并行执行
- 每个子代理给完整的上下文和指令
- 子代理完成后自动检查结果

### 阶段 3：自主调试

遇到问题时的处理流程：

1. **读错误信息** — 理解到底报了什么错
2. **分析原因** — 不猜，看代码和日志
3. **假设验证** — 列出可能原因，逐一排查
4. **修复** — 改代码，跑测试
5. **确认修复** — 测试通过才算完

**调试原则：**
- 最多重试 3 次相同方案，不行就换思路
- 不要盲目重试同样的操作
- 如果一个方案死活不通，换方案而不是硬刚
- 只有在穷尽所有自主排查手段后才向用户求助

### 阶段 4：验证交付

所有任务完成后：

1. **自动运行全部测试** — 不要求用户手动跑
2. **检查代码质量** — 审查自己写的代码
3. **确认规格覆盖** — 对照设计方案逐条检查
4. **生成完成报告** — 告诉用户做了什么、改了哪些文件、测试结果

报告格式：
```
## 完成报告

**实现内容：** [一句话总结]

**修改文件：**
- path/to/file1.py — [做了什么]
- path/to/file2.py — [做了什么]

**测试结果：** X 个测试全部通过

**提交记录：**
- abc1234 feat: ...
- def5678 feat: ...

**注意事项：** [如果有的话]
```

然后询问用户下一步（合并、PR、部署等）。

## 禁止行为

- **禁止频繁打断用户** — 能自己解决的自己解决
- **禁止偷懒跳步骤** — 每一步都要实际执行，不能说"你自己跑一下"
- **禁止甩锅** — 遇到错误自己调试，不要说"请你检查一下"
- **禁止假装完成** — 没跑通测试不能说"应该没问题"
- **禁止占位符** — 代码写完整，不留 TODO
- **禁止等待确认** — 除了上面列的"可以问"的情况外，直接往前推进

## 与 xgs-ideation 的衔接

当 `xgs-ideation` 完成设计并获得用户确认后：
- 用户同意继续 → 自动进入 autopilot 模式
- 从阶段 1 开始，一路跑到阶段 4 交付

## 核心原则

- **自主性** — 最大限度减少用户干预
- **完整性** — 每一步都要实际做，不偷懒
- **可靠性** — 自己跑验证，拿结果说话
- **韧性** — 遇到问题自己排查，不轻易放弃
- **透明性** — 最后给一份清晰的完成报告
